Why This Onboarding Guide Exists

The old problem

  • Dashboard was hardcoded to one team's AI classification data.
  • Every page imported the same data_loader.py.
  • Flat pages/ directory — no grouping.
  • Adding a page meant rewriting shared files and risking merge conflicts.

The new model

  • Each team owns a self-contained folder under dashboard/teams/.
  • A manifest.toml tells the dashboard what pages exist and where the data lives.
  • Nav auto-groups by section: AI Crypto Private Credit.
  • Drop a folder in, reload, it shows up. Remove it, it's gone. No shared-file edits.

Anatomy of a Team Folder

What lives inside

dashboard/teams/<your_slug>/ manifest.toml # declares section, sub-team, pages, data dir data_loader.py # your @st.cache_data readers pages/ __init__.py <page_a>.py <page_b>.py README.md # 10-line team intro

Your data goes in artifacts/<your_slug>/. Point the manifest at it.

What you never touch

  • dashboard/app.py — auto-discovers your team.
  • dashboard/core/ — framework code.
  • Other teams' folders.
  • src/ai_corpus/ — AI team's pipeline internals.
Rule of thumb: if you're editing a file outside your own dashboard/teams/<slug>/, stop and ask whether there's a manifest- or data-loader-level change that accomplishes the same thing.

The Manifest is the Contract

Everything the dashboard needs to know about your team lives in one TOML file.

[team] section = "Crypto" # "AI" | "Crypto" | "Private Credit" subteam = "Risk Analysis" # "Classification & Intent" | "Risk Analysis" display_name = "Crypto - Risk Analysis" owners = ["Your Name", "Teammate"] order = 2 # within the section [data] artifacts_dir = "artifacts/crypto_risk" # relative to repo root [[pages]] path = "pages/var_dashboard.py" title = "VaR Dashboard" icon = ":material/chart:" # optional Material icon order = 1 [[pages]] path = "pages/stress_tests.py" title = "Stress Tests" order = 2

Add a page? Add a [[pages]] block. Remove a page? Delete the block and the file. Reload Streamlit — that's it.

Adding a Page (The Easy Way)

One command does everything

python scripts/new_team_page.py \ --team crypto_risk \ --page-slug var_dashboard \ --title "VaR Dashboard" \ --icon ":material/chart:"

The scaffolder:

  • Validates your team slug against the registered teams.
  • Picks the next order automatically.
  • Appends a [[pages]] block to your manifest.toml.
  • Writes a starter page file with a missing-data guard.
  • Refuses to overwrite existing pages.
Reload check: streamlit run dashboard/app.py — new page appears under your section, title matches, no restart gymnastics.

The Data Loader Pattern

Each team owns its data_loader.py. Use the core path helper to resolve your artifacts directory from the manifest — never hard-code paths.

from __future__ import annotations import pandas as pd import streamlit as st from dashboard.core.paths import team_artifacts_dir @st.cache_data def load_var() -> pd.DataFrame: d = team_artifacts_dir(__file__) if d is None: return pd.DataFrame() path = d / "var_daily.csv" if not path.exists(): return pd.DataFrame() return pd.read_csv(path)
  • team_artifacts_dir(__file__) reads your manifest's [data].artifacts_dir.
  • Returns None if no data dir is declared yet.
  • Always return an empty DataFrame when data is missing — never fabricate.
  • Wrap every reader in @st.cache_data.
  • One loader function per artifact. Small, testable, cached.
  • Pages import your loaders by full dotted path.

A Complete Page in Under 20 Lines

import streamlit as st import pandas as pd from dashboard.core.ui import missing_data_warning from dashboard.teams.crypto_risk.data_loader import load_var st.title("VaR Dashboard") df = load_var() if df.empty: missing_data_warning( "Drop var_daily.csv into artifacts/crypto_risk/ and reload." ) st.metric("Latest 1-day VaR (99%)", f"${df.iloc[-1]['var_99']:.0f}") st.line_chart(df.set_index("date")[["var_95", "var_99"]]) st.dataframe(df.tail(10), use_container_width=True)
  • No st.set_page_config — the app owns that.
  • No sys.path hacking — the app has already wired imports.
  • missing_data_warning is a shared helper: warns and calls st.stop().

Bonus Pattern: Interactive Parameter Overrides

Got a scoring or aggregation formula with tunable weights? Expose them as sidebar controls so reviewers can stress-test assumptions without a pipeline rerun.

Reference: AI Classification weights

  • Two sliders (Maturity, Breadth) + derived Momentum — sum always 100%.
  • Reset button uses an on_click callback so session-state writes happen before widgets instantiate.
  • Pure recompute_scores(df, weights) function — unit-testable without Streamlit.
  • View-only: the stored CSV is never rewritten.

The blueprint

  1. Keep the formula pure — no Streamlit in the math.
  2. Write a weight_controls() or param_controls() sidebar widget.
  3. Persist state with a team-unique key_prefix.
  4. Always render a caption showing the applied values.
  5. Don't rewrite artifacts; apply overrides at view time only.

See dashboard/teams/ai_classification_intent/scoring.py.

Let an AI Agent Do the Wiring

Open a new Claude Code / Cursor / similar session in the repo and say:

"I'm on the Crypto Risk team. I have a CSV at artifacts/crypto_risk/var.csv. Add a page that shows the last 30 days as a line chart. Read AGENTS.md first."

The agent will:

  • Read AGENTS.md at the repo root for the full convention.
  • Open dashboard/teams/crypto_risk/ and inspect the manifest.
  • Scaffold the page via scripts/new_team_page.py (or equivalent).
  • Extend your team's data_loader.py with a cached reader.
  • Wire the page to your data and render the chart.
  • Tell you to run streamlit run dashboard/app.py to confirm.
Why this works: AGENTS.md encodes the whole decision tree — section, sub-team, data location, missing-data convention, reference implementation. The agent needs zero follow-up context.

Conventions That Keep Teams From Stepping On Each Other

Do

  • Keep your data under artifacts/<your_slug>/.
  • Use @st.cache_data on every reader.
  • Use a team-unique key_prefix for session-state and widget keys.
  • Call missing_data_warning(...) on empty frames.
  • Use Title Case for page titles; include units in axis labels.
  • Commit your team folder as a cohesive unit so merges stay local.

Don't

  • Don't call st.set_page_config inside a page — the app does.
  • Don't write to dashboard/core/ or dashboard/app.py.
  • Don't import from other teams' folders.
  • Don't fabricate placeholder numbers when data is missing.
  • Don't commit large raw data to the repo — use Git LFS or keep files local.
  • Don't assume a widget key is globally unique — prefix it.

Git Etiquette — Work On A Branch, Ship Via PR

Never push team changes straight to master. Every contribution goes through a pull request so we can catch regressions before they hit the shared dashboard.

The workflow

  1. Pull latest master.
  2. Branch off with a name like <your_slug>/<short-topic>.
  3. Keep your edits inside dashboard/teams/<your_slug>/ and artifacts/<your_slug>/.
  4. Commit in logical chunks with clear messages.
  5. Push your branch and open a pull request against master.
  6. Wait for review + CI. Do not self-merge.

The commands

git checkout master && git pull git checkout -b crypto_risk/var-dashboard # ... build your page, add your data ... python -m pytest tests/ -v streamlit run dashboard/app.py # smoke test locally git add dashboard/teams/crypto_risk \ artifacts/crypto_risk git commit -m "crypto_risk: add VaR dashboard page" git push -u origin crypto_risk/var-dashboard gh pr create --base master \ --title "crypto_risk: VaR dashboard" \ --body "Adds VaR page + loader. Local tests + streamlit green."
Why PRs matter here: six teams share one dashboard/app.py that discovers everyone's manifests. A malformed TOML or a broken import in your folder can break every other team's pages too. The PR is where we catch that — tests run, a reviewer spot-checks, and we confirm the dashboard still boots before anything lands on master.
